# Plan Integral de Gestión de Tareas y Optimización (Spartan Hub 2.0)

Este plan tiene como objetivo consolidar la infraestructura de IA recién implementada y resolver la deuda técnica crítica detectada, asegurando la estabilidad del sistema mediante controles de calidad rigurosos.

## 1. Análisis del Estado Actual

*   **Infraestructura IA (Backend/Frontend):**
    *   **Estado:** Completado ✅.
    *   **Logros:** Cascada de APIs con failover, Circuit Breakers, Caché en Ollama, Configuración Dinámica y Dashboard de Administración.
    *   **Acción Pendiente:** Validación final de integración E2E y limpieza de código (linting).

*   **Deuda Técnica Crítica (Backend):**
    *   **Estado:** Pendiente ⚠️.
    *   **Impacto:** Afecta la precisión de métricas biométricas (`trainingLoad` hardcodeado) y la persistencia de tokens de dispositivos (`updateDevice` no implementado).
    *   **Prioridad:** Alta (Bloqueante para funcionalidades core de salud).

*   **Base de Datos (PostgreSQL):**
    *   **Estado:** Incertidumbre ❓.
    *   **Riesgo:** Tests de conexión desactivados (`.skip`). Posibles regresiones no detectadas.

---

## 2. Plan de Ejecución Priorizado

### Fase 1: Consolidación y Calidad de IA (Inmediato)
*   **Objetivo:** Asegurar que el nuevo Dashboard y los servicios de IA cumplan con los estándares de calidad.
*   **Tareas:**
    1.  Ejecutar **Linting** en `spartan-hub/src` para validar `AiDashboard.tsx` y cambios en `App.tsx`/`Sidebar.tsx`.
    2.  Ejecutar **Tests de Backend** específicos de IA una vez más para confirmar estabilidad tras cambios en `.env.example`.
*   **Control de Calidad:** 0 errores de linting, 100% tests de IA pasando.
*   **Rollback:** Revertir commits de frontend si rompen el build.

### Fase 2: Resolución de Deuda Técnica Biométrica (Alta Prioridad)
*   **Objetivo:** Corregir cálculos hardcodeados y persistencia de tokens.
*   **Tareas:**
    1.  **Implementar `calculateTrainingLoad`**: En `biometricPersistenceService.ts`. Usar historial de actividades recientes (volumen * intensidad) para un cálculo real.
    2.  **Implementar `updateDevice`**: En `HealthConnectHubService` (o servicio equivalente) y conectarlo a `biometricController.ts` para guardar tokens refrescados.
*   **Validación:** Crear tests unitarios para `calculateTrainingLoad` y mocks para `updateDevice`.

### Fase 3: Reactivación de Infraestructura de Datos (Media Prioridad)
*   **Objetivo:** Validar integración con PostgreSQL.
*   **Tareas:**
    1.  Revisar configuración de Docker para PostgreSQL.
    2.  Quitar `.skip` de `postgresConnection.test.ts` y ejecutar.
    3.  Corregir configuración de Jest si persisten conflictos con `dist/`.

---

## 3. Protocolos y Metodología

*   **Revisión de Código:** Cada cambio se verifica con `npm run lint` y `npm test` antes de darlo por cerrado.
*   **Gestión de Errores:** Si un test falla, se detiene la fase actual hasta su resolución (fix-forward) o se revierte el cambio (rollback).
*   **Documentación:** Se actualizará el `README.md` o documentación interna con los nuevos servicios implementados.
